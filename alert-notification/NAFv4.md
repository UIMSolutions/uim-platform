# NAF v4 Architecture Views — Alert Notification Service

## NSOV-1 Service Overview

| Attribute | Value |
|---|---|
| Service Name | Alert Notification Service |
| Service ID | `uim-platform.alert-notification` |
| Version | 1.0.0 |
| Owner | UIM Platform Team |
| Runtime | D / vibe.d on Linux (Docker, Podman, Kubernetes) |
| Port | 8095 |
| Protocol | HTTP/REST (JSON) |
| Tenant Model | Multi-tenant via `X-Tenant-Id` header |

### Purpose

The Alert Notification Service provides a publish-subscribe routing mechanism for operational events generated by applications, services, and platform infrastructure. Producers POST events; the service evaluates configured routing rules (subscriptions + conditions) and dispatches to registered action endpoints.

---

## NSOV-2 Service Definitions

| Term | Definition |
|---|---|
| **Alert Event** | A structured notification emitted by a producer describing a condition change in a resource |
| **Condition** | A filter rule matching an event property (type, category, severity, resource attributes, tags) against a value using a predicate |
| **Action** | A delivery channel (EMAIL, SLACK, WEB_HOOK, STORE, PAGER_DUTY, JIRA, SERVICE_NOW, SIEM) invoked when a subscription matches |
| **Subscription** | Combines a list of conditions with a list of actions; enabled subscriptions are evaluated for every incoming event |
| **Matched Event** | An event persisted by a STORE action; retrievable by consumers via the pull API |
| **Undelivered Event** | An event for which action delivery failed; stored for auditing and retry |
| **Producer API** | REST endpoint accepting inbound event POSTs |
| **Consumer API** | REST endpoint allowing consumers to pull stored and undelivered events |
| **Management API** | REST endpoints for CRUD operations on conditions, actions, and subscriptions |

## C2 — Service Interfaces

| Interface | Direction | Protocol | Path |
|---|---|---|---|
| Producer API | Inbound | HTTP POST | `/api/v1/alert-notification/events` |
| Consumer API (matched) | Inbound | HTTP GET | `/api/v1/alert-notification/matched-events[/:id]` |
| Consumer API (undelivered) | Inbound | HTTP GET | `/api/v1/alert-notification/undelivered-events[/:id]` |
| Condition Mgmt | Inbound | HTTP REST | `/api/v1/alert-notification/conditions[/:id]` |
| Action Mgmt | Inbound | HTTP REST | `/api/v1/alert-notification/actions[/:id]` |
| Subscription Mgmt | Inbound | HTTP REST | `/api/v1/alert-notification/subscriptions[/:id]` |
| Health | Inbound | HTTP GET | `/api/v1/health` |
| WEB_HOOK delivery | Outbound | HTTP POST | Configurable URL per action |

## C5 — Security Considerations

| Concern | Mitigation |
|---|---|
| Tenant isolation | All data partitioned by `X-Tenant-Id` header; default tenant used when header absent |
| Non-root execution | Container runs as UID 1000 (`appuser`) |
| No secrets in image | Credentials passed via environment / Kubernetes Secrets |
| Input validation | DTO parsing rejects malformed JSON; enum conversion catches invalid values |
| Action endpoints | WEB_HOOK URLs should be validated against an allowlist in production |
| OWASP A01 (Access Control) | Multi-tenant repository isolation by design |
| OWASP A03 (Injection) | No dynamic SQL; structured JSON deserialization only |
